Crear el workflow de CI (Tests + SAST) ⏱ 12 min

Crear la carpeta .github/workflows/ en la raíz del proyecto y dentro el archivo ci.yml:

.github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    name: Tests + Análisis de Seguridad
    runs-on: ubuntu-latest

    steps:
      # 1. Descargar el código
      - name: Checkout código
        uses: actions/checkout@v4

      # 2. Instalar Python
      - name: Configurar Python 3.11
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      # 3. Instalar dependencias
      - name: Instalar dependencias
        run: pip install -r requirements.txt

      # 4. Ejecutar tests
      - name: Ejecutar tests
        run: pytest tests/ -v

      # 5. Análisis SAST con Bandit
      - name: 🔒 Análisis de seguridad (Bandit)
        run: |
          pip install bandit
          # -ll = reportar solo severidad MEDIA y ALTA
          # -ii = reportar solo confianza MEDIA y ALTA
          bandit -r . -ll -ii --exclude ./tests,./venv

Crear los 2 servidores en Render ⏱ 8 min

Vamos a crear dos servicios en Render: uno para Development y uno para Production.

🖥 Servidor Dev
  • Name: miapp-dev
  • URL: miapp-dev.onrender.com
  • Branch: main
  • Auto-Deploy: No (lo maneja Actions)
🚀 Servidor Prod
  • Name: miapp
  • URL: miapp.onrender.com
  • Branch: main
  • Auto-Deploy: No (lo maneja Actions)
  • Ir a render.com → Sign Up con tu cuenta de GitHub
  • New → Web Service → conectar el repositorio
  • Crear el servicio Dev: desactivar Auto-Deploy
  • Repetir y crear el servicio Prod: desactivar Auto-Deploy
  • Para cada servicio: Settings → Deploy Hook → copiar la URL
  • En GitHub: Settings → Secrets → Actions → agregar:
    RENDER_DEV_HOOK = URL del hook de Dev
    RENDER_PROD_HOOK = URL del hook de Prod
ℹ️

El Deploy Hook es una URL que cuando se llama con un POST, dispara un deploy en Render. GitHub Actions la llamará automáticamente cuando todo el CI pase.

Agregar CD al workflow (Deploy Dev → Smoke → Deploy Prod) ⏱ 12 min

Agregar los jobs de CD al archivo ci.yml (después del job test:):

.github/workflows/ci.yml — agregar estos jobs al final
  deploy-dev:
    name: Deploy a Desarrollo
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'

    steps:
      - name: 🚀 Deploy a Dev (Render)
        run: |
          curl -X POST ${{ secrets.RENDER_DEV_HOOK }}
          echo "Deploy a Dev disparado"

      - name: ⏳ Esperar que Render inicie
        run: sleep 30

  smoke-test:
    name: Smoke Test en Dev
    needs: deploy-dev
    runs-on: ubuntu-latest

    steps:
      - name: 🩺 Verificar que Dev responde
        run: |
          # Intentar 5 veces con 10 segundos entre intento
          for i in 1 2 3 4 5; do
            STATUS=$(curl -s -o /dev/null -w "%{http_code}" \
              https://miapp-dev.onrender.com/health)
            if [[ $STATUS -eq 200 ]]; then
              echo "✅ Dev responde correctamente (HTTP $STATUS)"
              exit 0
            fi
            echo "Intento $i fallido (HTTP $STATUS), reintentando..."
            sleep 10
          done
          echo "❌ Dev no responde después de 5 intentos"
          exit 1

  deploy-prod:
    name: Deploy a Producción
    needs: smoke-test
    runs-on: ubuntu-latest

    steps:
      - name: 🚀 Deploy a Prod (Render)
        run: |
          curl -X POST ${{ secrets.RENDER_PROD_HOOK }}
          echo "✅ Deploy a Producción disparado"
🔴

Reemplazar miapp-dev.onrender.com por la URL real de tu servicio Dev en Render. Cada grupo tiene una URL diferente.

✅

Concepto clave: La palabra needs: crea dependencias entre jobs. Si test falla, deploy-dev no corre. Si el smoke test falla, deploy-prod no corre. El código solo llega a Prod si superó todos los checks.
