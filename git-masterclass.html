01

¿Por qué el terminal sobre la GUI?

VSCode tiene una integración de Git muy conveniente. Pero en entornos de producción, pipelines CI/CD, servidores remotos, o cuando algo falla, solo tienes el terminal. Entender Git desde la línea de comandos te convierte en alguien que entiende qué pasa, no solo qué botón presionar.

Terminal

  • Disponible en cualquier servidor Linux/Mac
  • Scriptable — automatizable en CI/CD
  • Control total sobre cada operación
  • Mensajes de error completos y precisos
  • Más rápido con práctica

VSCode GUI

  • Visualización de diffs muy cómoda
  • Abstracciones que ocultan la complejidad
  • No disponible en servidores sin GUI
  • Errores a veces poco descriptivos
  • Depende del plugin/extensión
💡
TIP PARA EL AULA Usa VSCode normalmente para codear, pero abre el terminal integrado (Ctrl + `) para todos los comandos de Git. Así ves el código y el historial al mismo tiempo.
02

Configuración inicial de Git

Antes de usar Git, debes decirle quién eres. Esta información aparecerá en cada commit que hagas y es fundamental para la trazabilidad en equipos de desarrollo seguro.

bash 
# Identidad global (se aplica a todos los repos del sistema)
git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

# Editor por defecto (usamos VSCode como editor de mensajes)
git config --global core.editor "code --wait"

# Rama principal llamada 'main' (estándar moderno)
git config --global init.defaultBranch main

# Verificar configuración actual
git config --list
⚠️
SEGURIDAD: Nunca uses un email o nombre genérico compartido En equipos reales, cada commit debe ser rastreable a un autor real. Esto es parte de la cadena de responsabilidad (non-repudiation) en auditorías de software.

Verificar la instalación

bash 
git --version
# git version 2.43.0

# Ver el archivo de configuración global
cat ~/.gitconfig
03

Flujo básico: init, add, commit

El flujo fundamental de Git tiene tres áreas que debes entender: el Working Directory (tus archivos), el Staging Area (lo que preparas para commitear), y el Repository (el historial permanente).

Iniciar un repositorio

bash 
# Crear carpeta y entrar
mkdir mi-proyecto && cd mi-proyecto

# Inicializar git
git init
# Initialized empty Git repository in /mi-proyecto/.git/

# Ver el estado actual (siempre útil)
git status

Staging y commits

bash 
# Crear un archivo de ejemplo
echo "# Mi Proyecto Seguro" > README.md

# Ver qué cambió
git status
# Untracked files: README.md

# Agregar al staging area (un archivo específico)
git add README.md

# O agregar todo (con cuidado)
git add .

# Hacer el commit con mensaje descriptivo
git commit -m "feat: agregar README inicial del proyecto"

# Ver historial de commits
git log --oneline
✅
BUENA PRÁCTICA: Conventional Commits Usa prefijos como feat:, fix:, docs:, security:, refactor: en tus mensajes. Facilita los changelogs automáticos y la revisión de código en PRs.

¿Qué no commitear?

.gitignore 
# Crear .gitignore ANTES del primer commit

# Secretos y configuración sensible
.env
.env.local
*.key
secrets.json
config/credentials.yml

# Dependencias
node_modules/
vendor/

# Build artifacts
dist/
build/
*.class

# IDEs
.vscode/settings.json
.idea/
🚨
RIESGO DE SEGURIDAD CRÍTICO Un archivo .env con API keys o passwords commiteado a un repo público puede ser detectado por bots en segundos. GitHub Secret Scanning existe justamente por esto. Usa .gitignore siempre. Si ya commiteaste un secreto, debes rotarlo inmediatamente — la historia de Git es permanente.
04

Branches y merge

Las ramas son el corazón del trabajo colaborativo. Cada feature, bugfix o experimento va en su propia rama. La rama main siempre debe estar en un estado deployable y estable.

Crear y navegar ramas

bash 
# Ver todas las ramas
git branch

# Crear y cambiar a nueva rama (forma moderna)
git switch -c feature/login-seguro

# Forma clásica equivalente
git checkout -b feature/login-seguro

# Volver a main
git switch main

# Ver rama actual
git branch --show-current

Merge y resolución de conflictos

bash 
# Estando en main, integrar feature
git switch main
git merge feature/login-seguro

# Si hay conflictos, Git marca los archivos
# <<<<<<< HEAD
# tu versión
# =======
# versión de la otra rama
# >>>>>>> feature/login-seguro

# Después de resolver el conflicto manualmente:
git add archivo-resuelto.js
git commit -m "merge: resolver conflicto en login-seguro"

# Eliminar rama ya integrada
git branch -d feature/login-seguro
ℹ️
REBASE vs MERGE git rebase reescribe el historial para que sea lineal. Útil para mantener un historial limpio antes de abrir un PR. Regla de oro: nunca hagas rebase de ramas compartidas (como main). Solo en tu rama de feature, antes de mergearlo.
05

Trabajo con GitHub (remote)

Un remote es una copia de tu repositorio en un servidor externo. GitHub es la plataforma más usada. La autenticación con SSH es más segura que HTTPS con contraseña.

Configurar SSH key para GitHub

bash 
# Generar llave SSH (Ed25519 es el algoritmo recomendado)
ssh-keygen -t ed25519 -C "tu@email.com"

# Ver la llave pública para agregarla en GitHub
cat ~/.ssh/id_ed25519.pub

# Probar la conexión
ssh -T git@github.com
# Hi username! You've successfully authenticated.
🔑
NUNCA compartas tu llave privada El archivo id_ed25519 (sin .pub) es tu llave privada. La pública (.pub) es la que subes a GitHub. Usa una passphrase al generarla para protección adicional.

Conectar repo local con GitHub

bash 
# Agregar el remote (después de crear el repo en GitHub)
git remote add origin git@github.com:usuario/mi-proyecto.git

# Verificar remotes configurados
git remote -v

# Primer push: publicar la rama main
git push -u origin main

# Pushes siguientes (ya con upstream configurado)
git push

# Bajar cambios del remote
git pull

# Clonar un repositorio existente
git clone git@github.com:usuario/mi-proyecto.git

El flujo colaborativo completo

1
Crear rama local
git switch -c feature/mi-feature — nunca trabajes directamente en main.
2
Hacer commits pequeños y frecuentes
Commits atómicos son más fáciles de revisar y de revertir si algo falla.
3
Push de la rama al remote
git push origin feature/mi-feature — sube tu trabajo a GitHub.
4
Abrir un Pull Request en GitHub
Desde la web de GitHub, propones integrar tu rama en main y solicitas revisión.
5
Code Review y aprobaciones
Tus compañeros revisan el código. Con branch protection, se requieren al menos 2 aprobaciones.
6
Merge a main
Solo después de las aprobaciones requeridas. La rama de feature se elimina.
06

Pull Requests y Code Review

Un Pull Request (PR) es una propuesta formal de integración de código. Es el mecanismo central de revisión de calidad y seguridad en equipos profesionales. En Desarrollo Seguro, el code review es una línea de defensa contra vulnerabilidades.

Crear un PR desde el terminal

bash 
# Instalar GitHub CLI (opcional pero muy útil)
# https://cli.github.com

# Autenticarse
gh auth login

# Crear PR directamente desde el terminal
gh pr create --title "feat: implementar autenticación JWT segura" \
              --body "Implementa JWT con RS256. Closes #42" \
              --base main \
              --reviewer compañero1,compañero2

# Ver PRs abiertos
gh pr list

# Ver estado del PR actual
gh pr status
🔍
QUÉ REVISAR EN UN CODE REVIEW DE SEGURIDAD Busca: inputs sin validar, SQL sin parametrizar, secretos hardcodeados, uso inseguro de crypto, manejo incorrecto de errores que expone info sensible, permisos excesivos, dependencias con vulnerabilidades conocidas.
07

Branch Protection: 2 aprobaciones obligatorias

La protección de ramas es una control de seguridad de proceso que impide que código no revisado llegue a producción. Vamos a configurar main para requerir exactamente 2 aprobaciones antes de cualquier merge.

⚙️

GitHub → Settings → Branches → Branch protection rules

1
Ir a Settings del repositorio
En GitHub, entra a tu repo → pestaña Settings (debes ser admin/owner del repo).
2
Abrir "Branches" en el sidebar
En el menú izquierdo, busca la sección Code and automation → Branches.
3
Crear nueva regla
Click en "Add branch ruleset" o "Add classic branch protection rule".
4
Configurar el nombre de rama
En Branch name pattern escribe main. Puedes usar main|master o release/* para múltiples ramas.
5
Activar "Require a pull request before merging"
Marca este checkbox. Aparecerán sub-opciones. Activa "Require approvals" y cambia el número a 2.
6
Opciones adicionales recomendadas
✓ Dismiss stale reviews: si alguien pushea después de la aprobación, las aprobaciones se invalidan.
✓ Require review from Code Owners: útil si tienes un CODEOWNERS file.
✓ Require status checks: bloquea merge si los tests fallan.
7
Activar "Include administrators"
Muy importante: sin esto, los admins pueden saltarse las reglas. En equipos reales, nadie debe estar por encima del proceso.
8
Guardar con "Create" o "Save changes"
La regla entra en vigor inmediatamente. Prueba intentando hacer push directo a main — debe bloquearte.
⚠️
PROBAR QUE FUNCIONA Después de configurarlo, intenta: git push origin main directamente. GitHub debe rechazarlo con un error. Luego abre un PR y confirma que no se puede hacer merge sin las 2 aprobaciones.

Alternativa: Rulesets (GitHub moderno)

Los Rulesets son la versión moderna de las Branch Protection Rules. Permiten aplicar reglas a múltiples ramas con patrones y tienen más opciones de granularidad.

bash — verificar desde terminal 
# Con GitHub CLI puedes ver las reglas configuradas
gh api repos/{owner}/{repo}/branches/main/protection

# Resultado esperado (fragmento)
{
  "required_pull_request_reviews": {
    "required_approving_review_count": 2,
    "dismiss_stale_reviews": true
  },
  "enforce_admins": { "enabled": true }
}
🏛️
POR QUÉ ESTO IMPORTA EN DESARROLLO SEGURO Esta configuración implementa el principio de Separation of Duties en el ciclo de desarrollo. Ningún desarrollador —ni el más senior— puede introducir cambios sin revisión. Es equivalente a los controles de doble firma en operaciones financieras críticas. Muchos estándares como SOC 2, ISO 27001 y PCI-DSS requieren controles de este tipo en el SDLC.
